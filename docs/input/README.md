# 阶段一文档模板使用说明

本目录提供以下五类阶段一规划设计文档模板：

- 对标调研报告模板
- 功能需求设计模板
- 技术需求设计模板
- 数据需求设计模板
- 验收测试用例模板

## 1. 设计原则

这套模板基于 `docs/input` 中的输入材料和 `docs/language` 中的现有成品整理，并做了以下优化：

- 将“背景、问题、需求、结论”与“实现、数据、测试”分层，避免一份文档承担过多职责。
- 强化文档之间的追溯关系，确保每个需求都能追到功能点、技术方案、数据准备和测试用例。
- 默认适配 0.5 人天粒度拆解，便于后续直接生成阶段二开发工作项。
- 兼顾管理视角和执行视角，既能评审，也能作为研发输入。

## 2. 推荐编写顺序

1. 先写对标调研报告，明确问题、需求来源、优先级与对标结论。
2. 再写功能需求设计，把需求转成可开发的功能点和工作项。
3. 然后写技术需求设计，明确实现方案、边界、接口和风险。
4. 再写数据需求设计，补齐数据源、字段、样例和联调准备。
5. 最后写验收测试用例，建立 P0/P1 验收口径和追溯矩阵。

## 3. 五份文档之间的输入输出关系

| 文档         | 主要输入                                 | 主要输出                                   | 供谁使用             |
| ------------ | ---------------------------------------- | ------------------------------------------ | -------------------- |
| 对标调研报告 | 用户反馈、项目部需求、现状材料、竞品资料 | 需求清单、优先级、对标结论、范围建议       | 产品、项目、研发     |
| 功能需求设计 | 对标调研报告                             | 功能点清单、依赖关系、工作项明细           | 产品、研发、项目     |
| 技术需求设计 | 对标调研报告、功能需求设计               | 技术方案、模块改造点、接口设计、风险清单   | 架构、研发           |
| 数据需求设计 | 功能需求设计、技术需求设计               | 数据源清单、字段模型、样例数据、数据工作项 | 研发、测试、数据对接 |
| 验收测试用例 | 功能需求设计、数据需求设计、技术需求设计 | 测试用例集、前置条件、通过标准、追溯关系   | 测试、项目、研发     |

## 4. 编写约束

- 所有需求必须标注来源，至少区分用户反馈、项目部需求、现网问题、合规要求中的一种。
- 所有 P0 需求必须写明业务价值，不能只有“要做什么”，没有“为什么要做”。
- 功能工作项和数据工作项建议统一按 0.5 人天粒度拆分。
- 技术文档优先写“约束、方案、风险”，不要重复铺陈业务背景。
- 数据文档优先写“数据源、字段、样例、验收数据”，不要替代技术文档讲实现细节。
- 验收文档必须可执行，避免“体验良好”“基本正确”这类不可量化表述。
