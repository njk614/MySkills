# Skill 路由规则说明

本文档用于统一约束 OpenClaw 或其他上层宿主在以下三个 Skill 之间的路由逻辑：

- `ruisi-twinioc-envmetric-skill`
- `ruisi-twinioc-command-skill`
- `ruisi-twinioc-spacecount-skill`

目标不是替代各自 `SKILL.md` 的职责定义，而是在入口层提供一份统一、可执行的分流规则，避免多个 Skill 对同一句用户问题产生抢路由或误命中。

## 适用范围

本规则适用于以下场景：

- OpenClaw 按技能包自动选择 Skill
- 上层 Agent 需要在多个睿思 Skill 之间进行意图分发
- 用户一句话中同时包含查询、控制、预约或空间分析意图
- 需要降低 `ruisi-twinioc-command-skill` 的泛查询误命中

## 路由目标

### 1. 环境统计类

路由到 `ruisi-twinioc-envmetric-skill`。

适用问题类型：

- 查询温度、湿度、CO2、PM2.5、PM10、TVOC、甲醛、噪声、光照等环境指标
- 查询环境达标率、波动幅度、空气质量指数
- 查询环境指标趋势
- 查询温度传感器实时或历史数据
- 查询环境指标元数据、区域元数据

典型关键词：

- `环境`
- `温度`
- `湿度`
- `CO2`
- `PM2.5`
- `PM10`
- `TVOC`
- `甲醛`
- `噪声`
- `光照`
- `空气质量`
- `达标率`
- `趋势`
- `传感器`

明确排除：

- 设备控制
- 楼宇控制指令
- 空间预约
- 空间利用指数
- 告警处理动作

### 2. 控制执行类

路由到 `ruisi-twinioc-command-skill`。

适用问题类型：

- 场景交互控制
- 图层、主题、演示、告警处理
- 摄像头视频操作
- 灯光、温控器、对象聚焦、对象选中、对象搜索
- 查询后立即执行动作的复合请求

典型关键词：

- `打开`
- `关闭`
- `切换`
- `执行`
- `控制`
- `聚焦`
- `选中`
- `搜索`
- `播放`
- `暂停`
- `回放`
- `摄像头`
- `视频`
- `灯`
- `灯光`
- `温控器`
- `空调`
- `告警处理`

明确排除：

- 纯环境指标查询
- 纯空间预约查询
- 纯空间利用指数查询

说明：

- `ruisi-twinioc-command-skill` 中的 D01 查询能力，仅适用于场景内容、孪生体对象、摄像头列表、视频面板等与执行上下文强相关的查询。
- 只要用户问题核心是环境指标数值或空间利用指数，不应因为出现“查询”“看看”“现在怎么样”等泛化表达而优先路由到 command-skill。

### 3. 空间预定与空间指数类

路由到 `ruisi-twinioc-spacecount-skill`。

适用问题类型：

- 查询会议室或灵活工位是否可预约
- 查询可预约空间列表
- 查询预约状态
- 创建预约、改期预约、取消预约
- 查询空间利用指数、空间占用率、空置率评分、区域结果、整层结果
- 追问空间利用指数相关公式或口径

典型关键词：

- `预约`
- `预定`
- `能不能订`
- `可预约`
- `会议室空闲`
- `工位`
- `改期`
- `取消预约`
- `空间利用指数`
- `空间利用`
- `占用率`
- `空置率`
- `履约率`
- `区域结果`
- `整层结果`

明确排除：

- 环境舒适度指标查询
- 设备控制
- 告警动作

## 总体判定顺序

当用户问题只包含单一意图时，按以下顺序判定：

1. 先识别是否包含明确动作意图。
2. 再识别是否包含空间预约或空间利用指数意图。
3. 再识别是否包含环境指标意图。
4. 如果都不满足，再进入其他 Skill 或澄清流程。

推荐优先级如下：

1. `混合意图拆分`
2. `command-skill`
3. `spacecount-skill`
4. `envmetric-skill`
5. `澄清或其他 Skill`

说明：

- 这里的优先级不是说 command-skill 可以吞掉所有请求，而是说一旦出现明确动作词，应优先判定为控制类，除非同一句中还包含独立查询子任务，需要拆分执行。
- 对纯查询问题，不应仅因包含“查一下”“看一下”“现在怎么样”而进入 command-skill。

## 混合意图拆分规则

如果一句话中同时包含多个意图，不应强行只选一个 Skill，而应拆成多个子任务按顺序执行。

### 1. 环境查询 + 控制动作

示例：

- `先看一下大会议室温度，再打开空调`
- `查一下主场 CO2 达标率，不达标就打开新风`

处理方式：

1. 先调用 `ruisi-twinioc-envmetric-skill` 获取环境数据。
2. 再调用 `ruisi-twinioc-command-skill` 执行控制动作。
3. 如果第二步依赖第一步判断结果，则由上层编排决定是否继续执行。

### 2. 空间查询 + 控制动作

示例：

- `查一下哪个会议室空着，然后切到那个会议室的摄像头`
- `先看看现在有没有可预约会议室，再打开视频墙`

处理方式：

1. 先调用 `ruisi-twinioc-spacecount-skill` 完成空间查询。
2. 再调用 `ruisi-twinioc-command-skill` 完成控制动作。

### 3. 环境查询 + 空间查询

示例：

- `大会议室现在适合开会吗`
- `会议室现在情况怎么样`

处理方式：

- 如果用户更关心舒适度、空气质量、温度、噪声，路由到 `ruisi-twinioc-envmetric-skill`。
- 如果用户更关心是否空闲、能否预约、占用率、利用情况，路由到 `ruisi-twinioc-spacecount-skill`。
- 若句子无法判断，应先澄清，不建议直接赌一个 Skill。

## 高风险歧义问法

以下问法容易产生冲突，应优先澄清或结合上下文判定：

### 1. 泛化状态描述

- `大会议室现在怎么样`
- `主场情况如何`
- `这个区域表现怎么样`

建议：

- 若上文在谈环境指标，走 `ruisi-twinioc-envmetric-skill`
- 若上文在谈空间占用或预定，走 `ruisi-twinioc-spacecount-skill`
- 若上文在谈场景对象或控制上下文，走 `ruisi-twinioc-command-skill`
- 若无上下文，先澄清

### 2. “可用吗” 类问法

- `这个会议室可用吗`
- `大会议室能用吗`

建议：

- 若语义是能否预定，走 `ruisi-twinioc-spacecount-skill`
- 若语义是设备或场景能否操作，走 `ruisi-twinioc-command-skill`
- 若语义是环境是否达标，走 `ruisi-twinioc-envmetric-skill`

### 3. “适合吗” 类问法

- `这个会议室现在适合开会吗`

建议：

- 若判断依据是温度、CO2、噪声、光照，走 `ruisi-twinioc-envmetric-skill`
- 若判断依据是是否有空房、能否预约，走 `ruisi-twinioc-spacecount-skill`

## 对 OpenClaw 的建议实现

建议入口层按以下流程实现：

1. 先做意图切分，判断一句话是否包含多个并列子句或条件动作。
2. 如果存在多个意图，先拆分，再分别路由，不做单 Skill 强行承接。
3. 如果只有单一意图，按“动作词 > 空间词 > 环境词”的优先级匹配。
4. 对 `怎么样`、`情况如何`、`可用吗`、`适合吗` 这类高歧义问法，优先参考上下文；仍无法判断时先澄清。
5. 不要把 `ruisi-twinioc-command-skill` 的 D01 当成所有查询的兜底入口。

可参考以下伪代码：

```text
if has_multiple_intents(query):
    subtasks = split_intents(query)
    route_each(subtasks)
elif has_action_keywords(query):
    route(command-skill)
elif has_space_keywords(query):
    route(spacecount-skill)
elif has_env_keywords(query):
    route(envmetric-skill)
else:
    ask_for_clarification()
```

## 典型示例

### 应路由到 envmetric-skill

- `大会议室的温度达标率多少`
- `主场过去 24 小时 CO2 达标率趋势`
- `有哪些环境指标可以查询`
- `所有温度传感器的当前温度`

### 应路由到 command-skill

- `打开大会议室温控器`
- `切换到主场摄像头`
- `聚焦大会议室`
- `处理当前告警`

### 应路由到 spacecount-skill

- `明天下午两点帮我订一个会议室`
- `现在有哪些空闲会议室`
- `楼层20空间利用指数是多少`
- `固定工位区占用率怎么样`

### 应拆分为多个 Skill

- `先查大会议室温度，再打开空调`
- `查一下哪个会议室空着，然后切到那个会议室摄像头`
- `如果主场空气质量差，就打开新风`

## 与各 Skill 文档的关系

本文件提供的是统一入口层路由规则。

- 各 Skill 自身的 `SKILL.md` 仍应保留职责、边界和不适用场景。
- 如果未来某个 Skill 的范围变更，应同步更新本文件。
- 如果 OpenClaw 未来支持更细的技能触发权重，本文件可继续扩展为打分式路由策略。
