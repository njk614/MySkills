## 2. 文档目的

本文档用于明确 Twinioc Demo 中英文版改造的实现路径，说明需要调整的模块、接口、数据结构、部署关系和主要风险点，确保阶段二开发能够按统一技术方案推进。

## 3. 技术目标

### 3.1 总体目标

在不破坏现有中文 Demo 能力的前提下，为四类 skill、孪易产品、MCP 接口和 Demo 数据链路增加英文可用能力，并形成统一的双语 Demo 技术架构。

### 3.2 具体目标

- 保留现有指令编码、Skill 主流程和核心后端协议不变。
- 以现有 Demo 为技术基底做增量改造，不新建第二套独立技术架构。
- 通过 locale 或语言上下文控制用户可见文案输出语言。
- 让中文 Demo 摆脱真实 IoT 依赖，改由模拟数据和可控接口支撑。
- 让英文 Demo 基于整理后的中文 Demo 快速配置，不复制多套底层逻辑。
- 让孪易产品前后端接收英文指令时，执行效果与中文一致。
- 让 MCP 接口兼容中英文调用和返回，并补齐双语文档样例。

## 4. 现状技术基线

### 4.1 现有文档与代码基线

- docs/API-DESIGN.md：统一 API 设计草案。
- docs/openapi.yaml：接口与对象 schema 草案。
- 各 skill 自身 SKILL.md：说明层工作流基线。
- 当前中文 Demo 场景与调用链路：演示层基线。

### 4.2 现有实现现状

| 模块           | 现状说明                                  | 技术特点                           | 当前限制                             |
| -------------- | ----------------------------------------- | ---------------------------------- | ------------------------------------ |
| command        | 负责 NL 到 instruction_order 的生成与发送 | 运行时有中文计划文本与中文确认逻辑 | 英文计划文本与英文确认词尚未完整打通 |
| dataquery      | 负责温度和 MCP 只读查询                   | 返回文案主要中文                   | 英文输出和英文别名命中能力不足       |
| opeationrule   | 负责规则记录、规则匹配、pending 流程      | 规则解析正则主要中文               | 英文规则句式无法稳定解析             |
| spacecount     | 负责空间预定与空间指数查询                | 错误文案和说明文案主要中文         | 英文结果说明与英文建议不完整         |
| 中文 Demo      | 已可展示并联动现有中文链路                | 具备基础演示价值                   | 仍存在真实 IoT 调用依赖              |
| 模拟数据生成器 | 已有基础 mock 能力                        | 可生成部分演示数据                 | 与传感器和操作接口绑定不完整         |
| 孪易指令接收层 | 现以中文指令和中文对象名为主              | 与当前执行链路耦合较深             | 英文指令兼容性不足                   |
| MCP 接口       | 已有可调用接口和文档                      | 接口契约基础已具备                 | 双语调用与返回语义未统一             |

### 4.3 已识别技术约束

- instruction_order 编码不可变更，否则将影响现有 SendInstruction 流程。
- 孪易英文指令改造应优先通过归一化和映射层实现，不宜重写底层执行协议。
- Demo 必须优先脱离真实 IoT 依赖，不能以真实设备在线为前提。
- 模拟数据与操作接口必须保持一致，否则演示结果不可控。
- 双语能力必须优先兼容现有中文逻辑。
- 会议纪要和参考图只提供展示方向参考，技术方案不应因此引入大规模前端重构或场景资源重建。

## 5. 目标技术方案

### 5.1 总体方案

采用“共享执行层 + Demo 配置层 + 语言资源层 + 英文映射层 + 模拟数据层 + 双语接口层”的技术方案。

核心思想如下：

- Skill 执行层共享：继续复用现有四类 skill 主流程。
- Demo 配置层拆分：中文 Demo 先做去 IoT 化整理，再基于其配置英文 Demo，保留现有 Demo 主体结构。
- 输出层 locale 化：Skill、孪易返回、MCP 返回等用户可见文本按 locale 输出。
- 指令层归一化：英文指令先归一化，再映射到现有中文或标准执行对象。
- 数据层 mock 化：传感器和操作接口统一改由模拟数据生成器支撑。
- 接口层双语化：MCP 接口支持中英文调用和返回，文档同步补双语样例。

### 5.2 模块关系图

```text
用户输入
  -> Demo 入口层（中文 Demo / 英文 Demo）
  -> 技能触发层（SKILL.md / agent prompt / 孪易指令接收）
  -> 语言判断层（locale / language_context）
  -> 实体映射层（英文别名 -> 标准实体名）
  -> 执行层（command / dataquery / opeationrule / spacecount / 孪易执行）
  -> 数据层（模拟数据生成器 / mock 接口 / MCP）
  -> 文案渲染层（按 locale 输出）
  -> 用户返回
```

### 5.3 模块改造策略

#### Skill 层

- 保留现有四类 skill 的主流程。
- 增加英文说明层、英文文案、英文确认链路、英文规则解析和英文别名映射。

#### Demo 配置层

- 先梳理中文 Demo 中的孪生体和真实 IoT 调用依赖。
- 移除不必要真实调用，对必要调用统一改为 mock 接口。
- 基于整理后的中文 Demo 建立英文 Demo 配置、英文入口和英文展示信息。
- 若参考会议纪要和参考图做展示优化，应仅限现有页面信息呈现、文案和入口层面的轻量调整，不单独引入大规模 UI/UE 重做。

#### 模拟数据层

- 修改模拟数据生成器，覆盖传感器数据和操作回执。
- 让传感器查询和操作接口统一读取 mock 数据源。

#### 孪易前后端

- 增加英文命令归一化和对象别名映射。
- 让英文指令最终落到现有中文或标准执行对象，并保持执行效果一致。

#### MCP 接口层

- 兼容中英文调用参数。
- 兼容双语返回消息和错误说明。
- 调整接口文档和调用样例。

### 5.4 分阶段技术落地顺序

结合对标结论和功能需求拆解，阶段二技术实施建议按以下顺序推进：

| 阶段 | 技术主目标                           | 覆盖内容                                                  | 原因说明                                                  |
| ---- | ------------------------------------ | --------------------------------------------------------- | --------------------------------------------------------- |
| T1   | 中文 Demo 去 IoT 化并完成 mock 接管  | 中文 Demo、IoT 接入层、模拟数据生成器、操作接口           | 如果仍依赖真实设备，后续英文 Demo、MCP 联调和验收都不稳定 |
| T2   | 打通英文查询和英文执行主链路         | command、dataquery、语言资源层、实体映射层                | 这是 P0 演示闭环的最短路径，直接决定英文 Demo 是否可演示  |
| T3   | 建立英文 Demo 配置与孪易英文指令兼容 | Demo 配置层、孪易前后端、指令归一化映射层                 | 需要在主链路稳定后保证 Demo 层入口、展示和执行效果一致    |
| T4   | 完成 MCP 双语兼容与交付文档          | MCP 接口、docs/API-DESIGN.md、docs/openapi.yaml、调用样例 | 需要在执行链路和 mock 数据稳定后统一接口语义与样例        |
| T5   | 补齐规则与空间类增强能力             | opeationrule、spacecount                                  | 属于 P1 增强能力，应在主链路可交付后补齐                  |

阶段顺序说明：

- T1 对应功能需求中的 F-11、F-13，是所有可演示、可联调、可验收工作的技术前提。
- T2、T3、T4 共同构成 P0 技术主链路，对应功能需求中的 F-01 至 F-06、F-12、F-14、F-15。
- T5 对应功能需求中的 F-07、F-08、F-09，建议在主链路完成后并行补齐。

## 7. 接口与数据结构设计

### 7.1 接口设计原则

- 尽量不改动现有对外接口主结构。
- 新增语言字段和数据源字段优先采用兼容式可选字段。
- 英文命令优先通过归一化和映射层兼容，不直接改底层协议。
- 演示环境优先使用 mock 数据源，避免对真实 IoT 的硬依赖。

### 7.2 建议新增字段

| 对象         | 字段名           | 类型   | 是否必填 | 说明                  |
| ------------ | ---------------- | ------ | -------- | --------------------- |
| 通用请求     | locale           | string | 否       | 建议值为 zh-CN、en-US |
| 通用请求     | language_context | object | 否       | 用于补充语言判断来源  |
| 通用请求     | data_source_mode | string | 否       | 建议值为 mock、live   |
| 指令请求     | command_language | string | 否       | 标记原始指令语言      |
| 规则记录     | rule_language    | string | 否       | 标记规则原始语言      |
| 实体映射记录 | aliases          | array  | 否       | 英文别名列表          |

### 7.3 双语文案资源结构建议

```json
{
  "zh-CN": {
    "cancelled": "已取消操作",
    "no_pending": "当前没有待确认操作"
  },
  "en-US": {
    "cancelled": "Operation cancelled",
    "no_pending": "No pending confirmation found"
  }
}
```

### 7.4 英文别名映射结构建议

```json
{
  "entity_type": "device",
  "system_name": "Large Meeting Room",
  "display_name_en": "Large Meeting Room",
  "aliases_en": ["meeting room", "large meeting room"],
  "source": "scene_info"
}
```

### 7.5 Demo mock 绑定结构建议

```json
{
  "twin_id": "demo-device-001",
  "sensor_binding": "mock.temperature.large_meeting_room",
  "operation_binding": "mock.operation.light.large_meeting_room",
  "data_source_mode": "mock"
}
```

### 7.6 向后兼容要求

- 不改变现有指令编码和 SendInstruction 主链路。
- 对外接口新增字段必须采用兼容式可选字段，不影响现有中文调用方。
- 中文 Demo 在切换到 mock 数据后，展示和操作效果应保持与原演示口径一致。
- 英文规则解析后仍必须落到现有 parsed_rule 统一结构，避免下游执行逻辑分叉。
- 英文指令接收应通过映射层兼容，不得破坏现有中文命令效果。

### 7.7 验收与联调约束

- 演示环境默认使用 mock 数据源，除专项验证外不允许以真实 IoT 在线作为前置条件。
- 中文与英文请求应共享同一执行层和主要接口协议，不允许为英文单独复制一套核心 runtime。
- locale、command_language、rule_language、data_source_mode 等新增字段必须保持可选，并定义缺省行为。
- 英文规则、英文命令和英文查询在进入执行层前，都必须先归一化为现有标准对象或统一结构。
- docs/API-DESIGN.md、docs/openapi.yaml 和 Demo 调用样例需要与最终接口字段保持一致，避免文档与实现分叉。

## 8. 部署与模块关系

### 8.1 部署原则

- 继续沿用当前仓库内 skill 独立目录方式。
- Demo 配置、mock 数据、MCP 文档和英文入口尽量与现有中文实现共享底层逻辑。
- 演示环境默认使用 mock 数据源。

### 8.2 部署视图

```text
docs/API-DESIGN.md + docs/openapi.yaml
          |
          v
   Demo 配置层（中文 / 英文）
          |
          v
   技能与指令接收层
    |        |         |
 locale  aliases  command_mapping
    |        |         |
          v
   执行层 + MCP 层
          |
          v
   模拟数据生成器 / mock 接口
```

## 9. 技术风险与应对

| 风险编号 | 风险描述                         | 影响                   | 应对策略                                  |
| -------- | -------------------------------- | ---------------------- | ----------------------------------------- |
| T-01     | 只翻译 Skill，不处理 Demo 和接口 | 最终无法形成完整演示   | 明确将 Demo、mock、MCP 纳入 P0 范围       |
| T-02     | 中文 Demo 中真实 IoT 调用遗漏    | 演示环境不稳定         | 先做孪生体与 IoT 依赖清单，再逐项 mock 化 |
| T-03     | 模拟数据与操作接口状态不一致     | 演示结果失真           | 统一通过 mock 数据生成器生成读写结果      |
| T-04     | 英文指令解析结果与中文动作不一致 | 英文 Demo 效果失真     | 增加归一化和执行一致性校验                |
| T-05     | MCP 双语返回语义不统一           | 接入方无法稳定使用     | 明确 locale、错误文案和样例文档规范       |
| T-06     | 中英文共存引入中文回归           | 影响现有演示和项目使用 | 建立中文回归用例并作为 P0 验收项          |

## 10. 技术验收基线

| 序号 | 技术验收项           | 验收要求                                                             | 对应范围                                        |
| ---- | -------------------- | -------------------------------------------------------------------- | ----------------------------------------------- |
| 1    | 中文 Demo 去 IoT 化  | 演示环境在 mock 模式下可完成主要展示、查询和操作，不依赖真实设备在线 | 中文 Demo、IoT 接入层、模拟数据生成器           |
| 2    | 英文查询与执行主链路 | 英文输入可完成查询、计划生成、确认/取消、执行和结果返回              | command、dataquery、opeationrule                |
| 3    | 中英文共存           | 中文链路无回归，英文返回英文，中文返回中文，不出现主流程语言污染     | 四类 skill、孪易前后端、MCP                     |
| 4    | 英文 Demo 配置       | 英文 Demo 基于整理后的中文 Demo 配置生成，可稳定切换入口和展示信息   | Demo 配置层                                     |
| 5    | MCP 双语兼容         | 中英文请求和返回字段语义一致，文档与样例可直接交付联调               | MCP 接口、docs/API-DESIGN.md、docs/openapi.yaml |
| 6    | P1 增强能力          | 英文规则解析、英文实体别名映射、英文空间结果在主链路完成后可独立验收 | opeationrule、spacecount、实体映射层            |

## 11. 技术结论

- 不复制核心执行层脚本，优先共享 runtime 和执行逻辑。
- 会议纪要和两张参考图在技术上只形成展示方向约束，不形成新的系统重构目标。
- 中文 Demo 先去真实 IoT 依赖，再构建英文 Demo。
- 英文命令兼容优先通过映射层和归一化策略解决。
- 模拟数据生成器和 MCP 接口需要同步纳入双语改造范围。
- 只有 Skill、Demo、mock 数据和接口文档一起改，才能形成真正可交付的中英文 Demo。
